## 5. Thread-safety: `OpenRouterClient` (5 min)

**Problema:** Os contadores `self._call_count`, `self._total_input_tokens`, `self._total_output_tokens` são incrementados com `+=` simples. O `page_generator.py` usa `ThreadPoolExecutor` com até 30 workers — `+=` em Python **não é atômico**, o que causa perda silenciosa de contagem.

**Fix:**
```python
# core/openrouter_client.py
import threading

class OpenRouterClient:
    def __init__(self, ...):
        ...
        self._lock = threading.Lock()

    def generate_json(self, ...):
        ...
        with self._lock:
            self._call_count += 1
            if response.usage:
                self._total_input_tokens += response.usage.prompt_tokens
                self._total_output_tokens += response.usage.completion_tokens

    # Idem para generate_text()
```

## 10. Logging estruturado — substituir todos os `print()` (30 min)

**Problema:** O projeto inteiro usa `print()` com emojis para output. São **~45 print()** espalhados por 8 arquivos. Consequências:
- Não dá para filtrar severidade (info vs warning vs error)
- Não aparece em logs de produção rotacionados
- Sem timestamp automático
- Poluem output do pytest

**Fix — Criar:** `core/logger.py`

```python
import logging
import sys

def get_logger(name: str = "sitegen") -> logging.Logger:
    logger = logging.getLogger(name)
    if not logger.handlers:
        handler = logging.StreamHandler(sys.stdout)
        fmt = logging.Formatter(
            "%(asctime)s [%(levelname)s] %(message)s",
            datefmt="%H:%M:%S"
        )
        handler.setFormatter(fmt)
        logger.addHandler(handler)
        logger.setLevel(logging.INFO)
    return logger
```

Substituir em todos os módulos `core/`:
```python
# Antes:
print(f"  ✓ {len(keywords)} palavras-chave importadas")

# Depois:
from core.logger import get_logger
logger = get_logger()
logger.info("Palavras-chave importadas: %d", len(keywords))
```

> Os emojis podem ser mantidos no CLI (`generate.py`) para UX, mas o `core/` deve usar logging.

## 11. Exceções customizadas (20 min)

**Problema:** O código usa `ValueError`, `FileNotFoundError` e `Exception` genéricos. Quem faz `try/except` não consegue distinguir entre "config inválido" vs "API falhou" vs "template não encontrado".

**Fix — Criar:** `core/exceptions.py`

```python
class SiteGenError(Exception):
    """Base para todos os erros do SiteGen."""

class ConfigError(SiteGenError):
    """Config.yaml inválido ou campo obrigatório ausente."""

class APIError(SiteGenError):
    """Falha na comunicação com a API (OpenRouter/Gemini)."""

class TemplateError(SiteGenError):
    """Template não encontrado ou com placeholders não resolvidos."""

class ValidationError(SiteGenError):
    """Site gerado não passou na validação de qualidade."""
```

Substituir nos módulos:
- `config_loader.py`: `ValueError` → `ConfigError`
- `openrouter_client.py`: `ValueError` → `ConfigError` (chave ausente), exceções de API → `APIError`
- `template_injector.py`: `FileNotFoundError` → `TemplateError`
- `validator.py`: erros críticos → `ValidationError`
