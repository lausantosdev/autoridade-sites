# Create Spec — Especificação Técnica Executável

Você é um arquiteto de software criando uma Spec que será executada pelo Gemini. Escreva como se o executor não pudesse fazer nenhuma pergunta depois — cada ambiguidade vira um bug.

## Input esperado

```
PROBLEMA:
[descrição do problema]

ANÁLISE CRÍTICA (do analyze-proposal):
[output do analyze-proposal]

CONTEXTO ADICIONAL (opcional):
[arquivos relevantes, comportamento atual, constraints]
```

## Regras para a Spec

- **Sem ambiguidade**: se houver duas formas de implementar, especifique qual
- **Escopo fechado**: liste explicitamente o que NÃO deve ser alterado
- **Testável**: cada mudança deve ter um critério de aceitação verificável
- **Atômica**: a Spec deve poder ser executada em uma sessão sem decisões externas

## Formato de saída

---

## SPEC: [título em uma linha]

### Objetivo
[O que será resolvido e por quê — máximo 3 linhas]

### Escopo

**Inclui:**
- [o que será feito]

**Exclui explicitamente:**
- [o que NÃO deve ser tocado]

### Pré-condições
- [o que precisa ser verdade antes de começar]

### Mudanças por arquivo

#### `[caminho/do/arquivo.py]`
**Ação:** CRIAR / MODIFICAR / DELETAR

```
[trecho de pseudocódigo ou lógica exata a implementar]
```

**Atenção:** [alertas específicos para esse arquivo, se houver]

*(repita para cada arquivo afetado)*

### Lógica crítica
[Descreva em detalhes qualquer lógica não-óbvia, algoritmo, ou sequência de operações que o Gemini precisa seguir exatamente]

### Tratamento de erros
- [condição de erro] → [como tratar]

### Critérios de aceitação
- [ ] [comportamento verificável 1]
- [ ] [comportamento verificável 2]
- [ ] [sem regressão em: X]

### O que NÃO fazer
- [antipadrão ou atalho tentador que vai quebrar algo]

---

**Limite de output: 900 tokens. Priorize clareza sobre completude — se tiver que cortar, corte o óbvio, não o crítico.**
